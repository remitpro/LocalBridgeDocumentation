    Overview

    What is LocalBridge?

    LocalBridge is an enterprise-grade, self-hosted AI prototyping platform that enables teams to generate React/TypeScript code using local LLM models with complete data sovereignty.

    Key Responsibilities

    As an administrator, you'll manage: - System installation and configuration - User access and permissions - Security and compliance - Performance monitoring - Backups and disaster recovery - Updates and maintenance

    Installation & Setup

    Prerequisites

    System Requirements: - OS: Linux (Ubuntu 20.04+), macOS, or Windows with WSL2 - CPU: 8+ cores recommended - RAM: 16GB minimum, 32GB+ recommended - Storage: 100GB+ SSD - Network: Stable internet for initial setup

    Software Requirements: - Node.js 18+ and npm - PostgreSQL 14+ - Ollama (for AI models) - Git - Docker (optional, for containerized deployment)

    Step 1: Clone Repository

    git clone https://github.com/remitpro/LocalBridge.git
cd LocalBridge

    Step 2: Install Dependencies

    # Install Node.js dependencies
npm install

    Step 3: Database Setup

    # Create PostgreSQL database
createdb localbridge

# Set database URL
export DATABASE_URL="postgresql://username:password@localhost:5432/localbridge"

# Run migrations
npm run db:migrate

    Step 4: Install Ollama & Models

    # Install Ollama (Linux/macOS)
curl -fsSL https://ollama.ai/install.sh | sh

# Pull recommended models
ollama pull llama3.1:latest
ollama pull codellama:latest
ollama pull qwen2.5-coder:latest
ollama pull deepseek-coder:latest

# Verify installation
ollama list

    Step 5: Configure Environment

    Create .env file:

    # Database
DATABASE_URL=postgresql://username:password@localhost:5432/localbridge

# Authentication
JWT_SECRET=your-super-secret-jwt-key-change-this
SESSION_SECRET=your-super-secret-session-key-change-this

# Server
PORT=3001
NODE_ENV=production

# Frontend
VITE_API_URL=http://localhost:3001

# Optional: OAuth (Google)
GOOGLE_CLIENT_ID=your-google-client-id
GOOGLE_CLIENT_SECRET=your-google-client-secret
GOOGLE_CALLBACK_URL=http://localhost:3001/api/auth/google/callback

# Optional: OAuth (Microsoft)
MICROSOFT_CLIENT_ID=your-microsoft-client-id
MICROSOFT_CLIENT_SECRET=your-microsoft-client-secret
MICROSOFT_CALLBACK_URL=http://localhost:3001/api/auth/microsoft/callback

# Optional: GitHub Integration
GITHUB_CLIENT_ID=your-github-client-id
GITHUB_CLIENT_SECRET=your-github-client-secret
GITHUB_CALLBACK_URL=http://localhost:3001/api/auth/github/callback

# Optional: Analytics
VITE_ENABLE_ANALYTICS=true
VITE_ANALYTICS_ENDPOINT=https://your-analytics-endpoint.com

# Optional: Error Reporting
VITE_ENABLE_ERROR_REPORTING=true
VITE_ERROR_REPORTING_ENDPOINT=https://your-error-endpoint.com
VITE_APP_VERSION=1.0.0

    Step 6: Build & Start

    # Build for production
npm run build

# Start the application
npm start

    Step 7: Create Admin User

    # Access the database
psql $DATABASE_URL

# Create admin user
INSERT INTO users (email, password_hash, name, role, created_at, updated_at)
VALUES (
    'admin@yourcompany.com',
    '$2b$10$...', -- Use bcrypt to hash password
    'System Administrator',
    'admin',
    NOW(),
    NOW()
);

    Generate password hash: 

    const bcrypt = require('bcrypt');
const hash = await bcrypt.hash('your-password', 10);
console.log(hash);

    OAuth Configuration

    Google OAuth

    1. Go to Google Cloud Console
    2. Create a new project or select existing
    3. Enable Google+ API
    4. Create OAuth 2.0 credentials
    5. Add authorized redirect URI: http://your-domain.com/api/auth/google/callback
    6. Copy Client ID and Secret to .env

    Microsoft OAuth

    1. Go to Azure Portal
    2. Register a new application
    3. Add redirect URI: http://your-domain.com/api/auth/microsoft/callback
    4. Create client secret
    5. Copy Application ID and Secret to .env

    GitHub Integration

    1. Go to GitHub Developer Settings
    2. Create new OAuth App
    3. Set callback URL: http://your-domain.com/api/auth/github/callback
    4. Copy Client ID and Secret to .env

    User Management

    User Roles

    Role Permissions
    admin Full system access, user management, all features
    editor Create/edit projects, generate prototypes, GitHub integration
    viewer Read-only access to projects and prototypes
    user Basic access, limited features

    Creating Users

    Via Database

    INSERT INTO users (email, password_hash, name, role, created_at, updated_at)
VALUES (
    'user@company.com',
    '$2b$10$hashed_password',
    'User Name',
    'editor',
    NOW(),
    NOW()
);

    Via API (if admin panel exists)

    POST /api/admin/users
{
    "email": "user@company.com",
    "password": "secure-password",
    "name": "User Name",
    "role": "editor"
}

    Modifying User Roles

    UPDATE users 
SET role = 'admin' 
WHERE email = 'user@company.com';

    Deactivating Users

    UPDATE users 
SET deleted_at = NOW() 
WHERE email = 'user@company.com';

    Viewing Active Users

    SELECT id, email, name, role, created_at, last_login_at
FROM users
WHERE deleted_at IS NULL
ORDER BY created_at DESC;

    Security & Compliance

    Security Best Practices

    1. Strong Secrets

    # Generate secure secrets
openssl rand -base64 32  # For JWT_SECRET
openssl rand -base64 32  # For SESSION_SECRET

    2. HTTPS Configuration

    Use a reverse proxy (nginx/Apache) with SSL:

    server {
    listen 443 ssl http2;
    server_name localbridge.yourcompany.com;

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    location / {
        proxy_pass http://localhost:3001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
    }
}

    3. Firewall Rules

    # Allow only necessary ports
ufw allow 22/tcp    # SSH
ufw allow 80/tcp    # HTTP
ufw allow 443/tcp   # HTTPS
ufw deny 3001/tcp   # Block direct API access
ufw deny 5432/tcp   # Block direct DB access
ufw enable

    4. Database Security

    -- Create dedicated database user
CREATE USER localbridge_app WITH PASSWORD 'secure-password';
GRANT CONNECT ON DATABASE localbridge TO localbridge_app;
GRANT USAGE ON SCHEMA public TO localbridge_app;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO localbridge_app;

-- Revoke public access
REVOKE ALL ON DATABASE localbridge FROM PUBLIC;

    Audit Logging

    View Audit Logs

    SELECT 
    al.id,
    u.email as user_email,
    al.action,
    al.resource_type,
    al.resource_id,
    al.details,
    al.ip_address,
    al.created_at
FROM audit_logs al
LEFT JOIN users u ON al.user_id = u.id
ORDER BY al.created_at DESC
LIMIT 100;

    Common Audit Queries

    -- Failed login attempts
SELECT * FROM audit_logs 
WHERE action = 'login_failed' 
AND created_at > NOW() - INTERVAL '24 hours';

-- Prototype generations
SELECT * FROM audit_logs 
WHERE action = 'create_prototype' 
AND created_at > NOW() - INTERVAL '7 days';

-- Admin actions
SELECT * FROM audit_logs 
WHERE user_id IN (SELECT id FROM users WHERE role = 'admin')
ORDER BY created_at DESC;

    Monitoring & Maintenance

    Health Checks

    System Health

    # Check API health
curl http://localhost:3001/health

# Check database connection
psql $DATABASE_URL -c "SELECT 1;"

# Check Ollama
curl http://localhost:11434/api/tags

    Performance Metrics

    -- View generation metrics
SELECT 
    DATE(created_at) as date,
    COUNT(*) as total_generations,
    AVG(CAST(details->>'durationMs' AS INTEGER)) as avg_duration_ms,
    MAX(CAST(details->>'durationMs' AS INTEGER)) as max_duration_ms
FROM audit_logs
WHERE action = 'create_prototype'
GROUP BY DATE(created_at)
ORDER BY date DESC;

    Error Rates

    -- View error rates
SELECT 
    DATE(created_at) as date,
    COUNT(*) as total_errors
FROM audit_logs
WHERE action LIKE '%_failed'
GROUP BY DATE(created_at)
ORDER BY date DESC;

    Log Management

    View Application Logs

    # Real-time logs
tail -f logs/app.log

# Search for errors
grep "ERROR" logs/app.log

# View last 100 lines
tail -n 100 logs/app.log

    Log Rotation

    Create /etc/logrotate.d/localbridge:

    /path/to/localbridge/logs/*.log {
    daily
    rotate 30
    compress
    delaycompress
    notifempty
    create 0640 www-data www-data
    sharedscripts
    postrotate
        systemctl reload localbridge
    endscript
}

    Troubleshooting

    Common Issues

    Issue: Database Connection Failed

    Symptoms: - Application won't start - Error: "Connection refused"

    Solutions: 

    # Check PostgreSQL is running
systemctl status postgresql

# Start if needed
systemctl start postgresql

# Verify connection
psql $DATABASE_URL -c "SELECT 1;"

# Check credentials in .env

    Issue: Ollama Not Responding

    Symptoms: - Generation fails immediately - Error: "Failed to connect to Ollama"

    Solutions: 

    # Check Ollama status
systemctl status ollama

# Start if needed
systemctl start ollama

# Verify models
ollama list

# Test generation
ollama run llama3.1 "Hello"

    Issue: High Memory Usage

    Symptoms: - System slow - Out of memory errors

    Solutions: 

    # Check memory usage
free -h

# Check process memory
ps aux --sort=-%mem | head

# Restart Ollama (frees model memory)
systemctl restart ollama

# Consider upgrading RAM or using smaller models

    Issue: Slow Generation

    Symptoms: - Generation takes > 2 minutes - Timeouts

    Solutions: 1. Check CPU usage: top 2. Check if other processes consuming resources 3. Consider using faster model (Llama 3.1) 4. Increase timeout in configuration 5. Upgrade hardware

    Backup & Recovery

    Database Backups

    Manual Backup

    # Create backup
pg_dump $DATABASE_URL > backup_$(date +%Y%m%d_%H%M%S).sql

# Compress
gzip backup_*.sql

    Automated Backups

    Create /usr/local/bin/backup-localbridge.sh:

    #!/bin/bash
BACKUP_DIR="/var/backups/localbridge"
DATE=$(date +%Y%m%d_%H%M%S)

# Create backup directory
mkdir -p $BACKUP_DIR

# Backup database
pg_dump $DATABASE_URL | gzip > $BACKUP_DIR/db_$DATE.sql.gz

# Keep only last 30 days
find $BACKUP_DIR -name "db_*.sql.gz" -mtime +30 -delete

echo "Backup completed: db_$DATE.sql.gz"

    Add to crontab: 

    # Daily backup at 2 AM
0 2 * * * /usr/local/bin/backup-localbridge.sh

    Restore from Backup

    # Uncompress
gunzip backup_20260104_020000.sql.gz

# Restore
psql $DATABASE_URL < backup_20260104_020000.sql

    Disaster Recovery

    Recovery Plan

    1. Assess damage - Determine what was lost
    2. Restore database - From latest backup
    3. Verify data - Check critical tables
    4. Restart services - Bring system back online
    5. Notify users - Communicate status
    6. Post-mortem - Document and prevent recurrence

    Recovery Time Objective (RTO)

    • Target: < 4 hours
    • Database restore: ~ 30 minutes
    • Service restart: ~ 5 minutes
    • Verification: ~ 30 minutes

    Recovery Point Objective (RPO)

    • Target: < 24 hours
    • Daily backups: Maximum 24-hour data loss
    • Consider: More frequent backups for critical systems

    Useful SQL Queries

    -- User statistics
SELECT 
    role,
    COUNT(*) as count,
    COUNT(CASE WHEN last_login_at > NOW() - INTERVAL '30 days' THEN 1 END) as active_30d
FROM users
WHERE deleted_at IS NULL
GROUP BY role;

-- Project statistics
SELECT 
    COUNT(*) as total_projects,
    COUNT(CASE WHEN created_at > NOW() - INTERVAL '30 days' THEN 1 END) as new_30d
FROM projects
WHERE deleted_at IS NULL;

-- Generation statistics
SELECT 
    DATE(created_at) as date,
    COUNT(*) as generations,
    COUNT(DISTINCT user_id) as unique_users
FROM prototypes
GROUP BY DATE(created_at)
ORDER BY date DESC
LIMIT 30;
