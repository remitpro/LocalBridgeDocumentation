    üéØ Simplified Architecture: One Project = One Prototype

    ‚úÖ Strategic Decision

    Change: From "Project ‚Üí Multiple Prototypes" to "Project = Single Prototype"

    Rationale: Better UX for iterative development and continuous improvement

    üìä Current vs Proposed Architecture

    Current Architecture ‚ùå

    Project
  ‚îú‚îÄ‚îÄ Prototype 1
  ‚îú‚îÄ‚îÄ Prototype 2
  ‚îú‚îÄ‚îÄ Prototype 3
  ‚îî‚îÄ‚îÄ Prototype 4

Issues:
- Confusing hierarchy
- Context switching between prototypes
- Unclear which prototype to enhance
- Conversation history scattered

    Proposed Architecture ‚úÖ

    Project (IS the Prototype)
  ‚îú‚îÄ‚îÄ Version 1 (Initial generation)
  ‚îú‚îÄ‚îÄ Version 2 (Enhanced)
  ‚îú‚îÄ‚îÄ Version 3 (Further refined)
  ‚îî‚îÄ‚îÄ Version 4 (Final)

Benefits:
- Single focus
- Clear conversation flow
- All iterations in one place
- Natural enhancement workflow

    üéØ User Experience Improvements

    Before (Multiple Prototypes)

    User Journey:
1. Create project "Dashboard"
2. Generate prototype "Dashboard v1"
3. Want to improve? ‚Üí Create NEW prototype
4. Lose context
5. Confusing which is latest
6. Hard to compare versions

    After (Single Prototype)

    User Journey:
1. Create project "Dashboard"
2. Describe what you want
3. AI generates initial version
4. Continue prompting to improve
5. All versions tracked automatically
6. Clear progression timeline

    üöÄ Implementation Plan

    Phase 1: Database Migration

    Step 1.1: Add Conversation Table

    CREATE TABLE conversation_messages (
  id UUID PRIMARY KEY,
  project_id UUID REFERENCES projects(id),
  type VARCHAR(10) NOT NULL, -- 'user' or 'ai'
  content TEXT NOT NULL,
  version_number INTEGER,
  timestamp TIMESTAMP DEFAULT NOW()
);

    Step 1.2: Simplify Prototype Versions

    -- Link versions directly to projects
ALTER TABLE prototype_versions 
  ADD COLUMN project_id UUID REFERENCES projects(id);

-- Migrate existing data
UPDATE prototype_versions pv
SET project_id = p.project_id
FROM prototypes p
WHERE pv.prototype_id = p.id;

-- Eventually deprecate prototypes table
-- (Keep for backward compatibility initially)

    üîÑ Migration Strategy

    For Existing Users

    Option 1: Automatic Migration

    For each existing project with multiple prototypes:
1. Keep most recent prototype as main
2. Convert others to "archived projects"
3. Notify user of change
4. Provide "restore" option

    Option 2: User Choice

    Show migration wizard:
1. "You have 3 prototypes in 'Dashboard' project"
2. "Choose which to keep as main:"
   - Prototype A (most recent)
   - Prototype B
   - Prototype C
3. "Others will become separate projects"
4. User selects
5. Migration completes

    Option 3: Gradual Transition

    1. Keep old structure for existing projects
2. New projects use new structure
3. Provide "Upgrade to new format" button
4. Eventually deprecate old structure

    Recommended: Option 3 (Gradual Transition)
