    Database Integration Guide

    This guide explains how to set up and use the PostgreSQL database for persistent storage.

    Overview

    The application now uses PostgreSQL (via Neon) for storing: - Users: User accounts (optional, for multi-user support) - Conversations: Chat conversation metadata - Messages: Individual messages within conversations

    Architecture

    Frontend (React + Vite)
     HTTP/REST
Backend (Express + Node.js)
     SQL
Database (PostgreSQL via Neon)

    Setup Instructions

    1. Install Dependencies

    npm install

    This will install: - drizzle-orm - Type-safe ORM - @neondatabase/serverless - Neon PostgreSQL driver - express - Backend server - cors, helmet, express-rate-limit - Security middleware - ws - WebSocket support for Neon

    2. Configure Environment Variables

    The database connection string is already configured in your .env file:

    DATABASE_URL=postgresql://neondb_owner:npg_DwYLAyv65hWg@ep-cold-dust-abvtnhea-pooler.eu-west-2.aws.neon.tech/neondb?sslmode=require&channel_binding=require

    Note: This connection string is NOT exposed to the browser. It's only used by the backend server.

    3. Generate Database Migrations

    npm run db:generate

    This creates migration files in the drizzle/ directory based on your schema.

    4. Run Migrations

    npm run db:migrate

    This applies the migrations to your database, creating the necessary tables.

    5. Start the Application

    # Development mode (runs both frontend and backend)
npm run dev

# Or run separately:
npm run dev:client  # Frontend only (port 8080)
npm run dev:server  # Backend only (port 3001)

    Database Schema

    Users Table

    CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT UNIQUE,
  name TEXT,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

    Conversations Table

    CREATE TABLE conversations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  model TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

    Messages Table

    CREATE TABLE messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  conversation_id UUID REFERENCES conversations(id) ON DELETE CASCADE,
  role TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system')),
  content TEXT NOT NULL,
  metadata JSONB,
  created_at TIMESTAMP DEFAULT NOW()
);

    API Endpoints

    Conversations

    Method Endpoint Description
    GET /api/conversations Get all conversations
    GET /api/conversations/:id Get conversation with messages
    POST /api/conversations Create new conversation
    PATCH /api/conversations/:id Update conversation title
    DELETE /api/conversations/:id Delete conversation

    Messages

    Method Endpoint Description
    GET /api/messages/:conversationId Get all messages for conversation
    POST /api/messages Create new message
    DELETE /api/messages/:id Delete message

    Example API Usage

    Create a Conversation

    const response = await fetch('http://localhost:3001/api/conversations', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    title: 'My Chat',
    model: 'llama3.1:latest',
  }),
});

const conversation = await response.json();

    Add a Message

    const response = await fetch('http://localhost:3001/api/messages', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    conversationId: conversation.id,
    role: 'user',
    content: 'Hello, AI!',
  }),
});

const message = await response.json();

    Database Management

    View Database (Drizzle Studio)

    npm run db:studio

    This opens a web-based database browser at https://local.drizzle.studio

    Push Schema Changes (Development)

    npm run db:push

    This pushes schema changes directly to the database without creating migrations (useful for rapid development).

    Generate New Migrations

    After modifying server/db/schema.ts:

    npm run db:generate
npm run db:migrate

    Migration from localStorage

    The application previously used localStorage for storing conversations. To migrate:

    1. Export existing data (if needed): 

      const data = localStorage.getItem('ollama-chat-conversations');
console.log(JSON.parse(data));

    2. Import to database using the API endpoints above

    3. Clear localStorage (optional): 

      localStorage.removeItem('ollama-chat-conversations');

    Security Considerations

    Environment Variables

    • DATABASE_URL is NOT prefixed with VITE_ - it's server-side only
    • Never commit .env file to version control
    • Use .env.example as a template

    API Security

    • Rate limiting: 100 requests per 15 minutes per IP
    • CORS configured for frontend origin only
    • Helmet.js for security headers
    • Input validation with Zod
    • SQL injection protection via Drizzle ORM

    Database Security

    • SSL/TLS required for connections
    • Parameterized queries (no SQL injection)
    • Foreign key constraints for data integrity
    • Cascade deletes for cleanup

    Production Deployment

    Environment Variables

    Update .env.production.local:

    DATABASE_URL=your_production_database_url
VITE_API_URL=https://api.your-domain.com/api
PORT=3001
CORS_ORIGIN=https://your-domain.com
NODE_ENV=production

    Build and Deploy

    # Build both frontend and backend
npm run build

# Start production server
npm start

    Docker Deployment

    The Dockerfile has been updated to support the backend server. See DEPLOYMENT.md for details.

    Troubleshooting

    Connection Errors

    Error: Cannot connect to database

    Solution: 1. Verify DATABASE_URL in .env 2. Check network connectivity 3. Ensure Neon database is active

    Migration Errors

    Error: Migration failed

    Solution: 1. Check database permissions 2. Verify schema syntax 3. Review migration files in drizzle/ directory

    CORS Errors

    Error: CORS policy blocked

    Solution: 1. Verify CORS_ORIGIN matches your frontend URL 2. Check that backend server is running 3. Ensure VITE_API_URL is correct

    Performance Optimization

    Indexes

    The schema includes indexes on: - users.email - Fast user lookups - conversations.user_id - Fast user conversations - conversations.created_at - Sorted listings - messages.conversation_id - Fast message retrieval - messages.created_at - Message ordering

    Connection Pooling

    Neon uses connection pooling automatically for optimal performance.

    Caching

    Consider implementing Redis caching for frequently accessed data in production.

    Monitoring

    Database Metrics

    Monitor in Neon dashboard: - Connection count - Query performance - Storage usage - Error rates

    API Metrics

    Monitor: - Response times - Error rates - Request volume - Rate limit hits

    Backup and Recovery

    Automated Backups

    Neon provides automated backups. Configure in your Neon dashboard.

    Manual Backup

    # Export database
pg_dump $DATABASE_URL > backup.sql

# Restore database
psql $DATABASE_URL < backup.sql
